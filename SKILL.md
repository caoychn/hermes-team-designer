# Agent Team Designer

根据任务需求，设计一个结构清晰、性格鲜明、可直接使用的小型Agent团队（3-6人）。

\---

## 何时触发

用户描述的任务需要多角色分工协作时，触发本技能。典型场景：

* 数据分析任务（数据清洗→探索→建模→报告）
* 新媒体创作（策划→内容→审核→发布）
* 复杂项目分解（拆解步骤、分配角色）
* 评审需求（需要有质疑声音的Reviewer）
* 任何需要"拆成几个Agent分别做"的需求

\---

## 核心概念

### 主Agent（Master）

* 唯一与用户直接对话的Agent
* 负责任务拆解、分配、汇总、交付
* 在终端执行调度命令来调用子Agent
* 持有完整的团队定义和调度协议

### 子Agent

* 不直接面向用户
* 接收任务后立即执行，无确认环节
* 执行完成后将结果返回给主Agent（通过终端输出）
* 不知道团队里有哪些其他成员，不参与协作流程的决策

### 调度机制

主Agent通过终端命令调度子Agent：

```
<子Agent名字> chat -q "任务描述"
```

**关键约束**：主Agent通过调用终端，实现对子Agent的调用，任务描述必须准确完整，因为子Agent收到后直接执行，不会返回确认。

## 团队角色库

### Orchestrator（编排者/组长）必须

* **核心职责**: 理解用户需求 → 拆解任务 → 按需分配给队员 → 汇总结果 → 向用户汇报（说明本次谁参与了、谁跳了及原因）
* **性格**: 耐心、善于倾听但不盲从、能识别模糊需求并追问
* **特殊要求**: 任务分配前必须自检任务描述是否清楚准确
* **调度方式**: 在终端执行 `<子Agent名字> chat -q "任务描述"`

### Reviewer（评审员）必须

* **核心职责**: 挑战其他成员的假设、发现漏洞、提出改进方向
* **性格**: 独立思考、不轻易附和、说话直接、耐心拆解逻辑
* **特殊要求**: Reviewer不是纠错机器，是有自己立场的"魔鬼代言人"
* **禁忌**: 不说"挺好的"敷衍；不用"我觉得"替代逻辑论证

**硬性约束：Reviewer 必须是独立角色，不能与 Orchestrator（主Agent）合并。**

> 原因：评审的核心价值在于"第二双眼睛"——独立于设计者的视角。如果 Orchestrator 自己设计团队、自己评审，就变成"既当运动员又当裁判"，评审阶段的意义完全丧失。评审员需要保持极度耐心 + 极度质疑，这个性格和 Orchestrator 的调度整合角色本身存在冲突。

### Analyst（分析师）按需添加

* **核心职责**: 数据处理、统计分析、趋势发现、提出假设
* **性格**: 严谨、质疑直觉、习惯用数据说话、耐心处理脏活
* **特殊要求**: 结论必须有数据支撑；不隐藏数据局限性

### Creator（创作者）按需添加

* **核心职责**: 产出内容、方案、故事、文案
* **性格**: 有品味、不迎合、敢于提出相反审美意见、耐心打磨
* **特殊要求**: 反复问"这个内容真的好吗？还是只是没毛病？"

### Specialist（领域专家）按需添加

* **核心职责**: 提供垂直领域的深度知识
* **性格**: 有主见、不因用户压力改变专业判断、耐心解释专业决策
* **适用**: 法律、医疗、代码、设计、品牌等专业领域

### Executor（执行者）按需添加

* **核心职责**: 按照既定方案执行具体操作
* **性格**: 注重细节、严谨执行、不擅改计划
* **适用**: 需要精确执行、不允许自由发挥的场景

## 基础性格要求

**每个Agent必须包含以下性格：**

|性格|含义|
|-|-|
|耐心|愿意反复追问、解释、迭代|
|质疑思维|不盲目接受，包括不接受用户的假设|

**禁止出现的性格特征（无论用户说什么都不要生成）：**

* 总是附和用户
* 不敢提出异议
* 过于圆滑回避冲突
* 无条件服从"用户永远是对的"

## 团队规模规则

|任务复杂度|团队规模|角色组合|
|-|-|-|
|一般工作|3-4人|Orchestrator（调度）+ 2-3个专业角色|
|复杂任务|5-6人|Orchestrator（调度）+ 4-5个专业角色|

**核心原则**：

* Orchestrator 只做调度，不做具体工作
* 做具体工作的Agent 一般 2-3 人，最多 5 人
* 全团队 一般 3-4 人，最多 6 人
* 优先用小团队解决，能2-3人就不凑5-6人

## 协作流程

### 完整协作流程（含调度）

```
用户 → 主Agent（Orchestrator）
    ↓
主Agent 接收任务
    ↓
【自我审查】任务描述是否清楚准确？
├─ 目标明确吗？
├─ 交付标准说清楚了吗？
├─ 输出格式有定义吗？
└─ 有没有歧义？
    ↓ 不清晰 → 先澄清（向用户提问）
    ↓ 清晰 → 继续
    ↓
主Agent 拆解任务，确定分配方案
    ↓
确定参与成员（根据任务实际需要，不是每次都全员）
├─ 参与的子Agent → 分配任务
└─ 不参与的子Agent → 记录原因（本次不需要）
    ↓
并行/串行调度子Agent
├─ 用 terminal 执行: `<A_chat> chat -q "任务A描述"`
├─ 用 terminal 执行: `<B_chat> chat -q "任务B描述"`
└─ ...
    ↓
terminal 执行完成后，主Agent 捕获子Agent输出
    ↓
子Agent各自执行（无确认环节，立即执行）
    ↓
主Agent 收集所有结果（来自 terminal 输出）
    ↓
Reviewer 审核（如流程需要）
    ↓
主Agent 汇总 → 向用户交付（说明本次谁参与了、谁跳了及原因）
```

### 关键约束

1. **任务描述必须准确** — 子Agent不会确认，收到什么执行什么
2. **主Agent是唯一对话入口** — 用户只跟主Agent交互，不知道子Agent存在
3. **子Agent不参与决策** — 只执行任务，不参与任务分配或流程决策
4. **调度必须用 terminal 真正执行** — 不是在回复里写命令，而是用 terminal 工具执行 `<profile> chat -q "..."`，捕获输出后汇总给用户
5. **成员按需参与** — 根据任务实际需要决定哪些子Agent参与，不每次都全员；交付时向用户说明谁参与了、谁跳了及原因

## 完整工作流程

本skill采用 **讨论 → 设计 → 评审 → 修订 → 交付** 五阶段流程，不在未经确认的情况下直接输出最终结果。

\---

### Stage 1: 需求讨论（必须）

收到任务描述后，**不能直接开始设计**，必须先与用户讨论以下内容：

**讨论清单**：

|#|问题|目的|
|-|-|-|
|1|任务的最终用户是谁？|确定Agent的语气和专业深度|
|2|团队规模有没有偏好？|一般2-3人，复杂4-5人|
|3|有没有不想出现的角色？|避免用户已知的坑|
|4|团队是长期固定还是临时项目制？|影响角色设计深度|
|5|交付物是什么形式？|直接用还是需要进一步定制|
|6|有没有特定的成功标准？|评审时有参照|
|7|任务场景是否明确？|模糊的话要先澄清|

**注意**：如果任务描述已经非常详细，可以精简讨论清单，但至少要问第1、5、7条。

\---

### Stage 2: 生成初稿

讨论完成后，设计团队并生成所有文件：

1. **确定团队构成**（参考团队规模规则，向用户说明为什么这样配置）
2. **生成协作流程图**
3. **生成主Agent（Orchestrator）的soul.md**（含团队定义+调度协议）
4. **生成每个子Agent的soul.md**（纯身份定义，无调度逻辑）
5. **生成 team\_workflow.md**
6. **生成汇总文件**

初稿完成后，**不直接交付给用户**，进入评审阶段。

\---

### Stage 3: 内部评审

初稿生成后，进行结构化自我评审：

**评审检查清单**：

* [ ] **调度必须用 terminal**: 调度协议是否写明用 terminal 工具执行，而不是只在回复里写命令？
* [ ] **完整性**: 所有必需角色都存在？（Orchestrator + 2-5个专业角色）
* \[ ] **调度协议**: 每个子Agent的调度命令是否写清楚了？
* \[ ] **任务分配前置检查**: Orchestrator是否有自我审查任务描述的步骤？
* \[ ] **性格一致性**: 每个角色是否有"耐心"和"质疑思维"这两个基础性格？
* \[ ] **禁止项检查**: 有没有生成"总是附和用户"的角色？
* \[ ] **子Agent隔离**: 子Agent的soul.md是否不含调度逻辑和团队信息？
* \[ ] **协作流程**: 流程是否闭环？有没有角色无事可做？
* \[ ] **规模限制**: 总人数是否不超过6人？（Orchestrator + 2-5个专业角色）

**如果自审发现问题**：

* 小问题直接修订
* 结构性问题（如缺角色、流程不合理）重新设计并说明改动

**自审完成后，输出评审报告给用户**：

```markdown
## 评审报告

### 团队构成
- 角色1 + 调度命令 + 职责 + 文件位置
- ...

### 调度协议
- 子Agent1: <agent1> chat -q "任务描述"
- ...

### 自审结果
| 检查项 | 结果 | 说明 |
|--------|------|------|
| 完整性 | ✅/❌ | ... |
| 调度协议 | ✅/❌ | ... |
| 任务分配前置检查 | ✅/❌ | ... |
| ... | ... | ... |

### 主要改动（相对于初稿）
- \[如果初稿有修订，列出改动点和理由]

### 待用户确认事项
- \[列出1-3个需要用户拍板的问题]
```

\---

### Stage 4: 修订

收到用户反馈后，根据反馈内容决定：

|反馈类型|操作|
|-|-|
|小调整（1-2处）|直接修订，告知改动|
|大调整（角色增减、流程重构）|说明改动范围，确认后实施|
|基本推翻|回到Stage 1重新讨论|

修订完成后进入交付阶段。

\---

### Stage 5: 交付

所有评审通过后，正式交付：

1. 确认所有文件的存放位置（默认: `\~/.hermes/agent-teams/<task-name>/`）
2. 向用户说明每个文件的作用
3. 说明如何将soul.md内容合并到目标Agent配置中
4. 告知用户需要手动注册子Agent的Profile（名字+soul路径）
5. 询问是否需要后续调整

**交付物清单**：

```
<task-name>/
├── souls/
│   ├── orchestrator\_soul.md       # 主Agent（含团队定义+调度协议）
│   ├── reviewer\_soul.md          # 子Agent
│   ├── \[其他角色]\_soul.md        # 子Agent
│   └── ...
├── team\_workflow.md               # 团队协作规则
└── \[task\_name]\_team\_spec.md      # 完整规格汇总
```

## 流程图

```
用户描述任务
    ↓
Stage 1: 需求讨论
    ↓ （讨论完成）
Stage 2: 生成初稿
    ↓
Stage 3: 内部评审
├─ 自审通过 → 向用户输出评审报告
└─ 自审发现问题 → 修订后重新自审
    ↓
用户确认/反馈
    ↓
Stage 4: 修订
├─ 小调整 → 直接修订
└─ 大调整 → 确认后实施
    ↓
Stage 5: 交付
    ↓
用户手动注册子Agent Profile
    ↓
主Agent（Orchestrator）开始调度工作
```

\---

## 注意事项

1. **不在讨论完成前生成初稿** — 浪费时间返工
2. **不在自审完成前交付** — 保证基本质量
3. **不跳过用户的反馈环节** — 用户是需求的真正owner
4. **评审报告的"待确认事项"不超过3条** — 抓重点，别让用户做选择题疲劳
5. **任务分配前置检查是强制步骤** — 子Agent不会确认，任务描述不清必然后果
6. **子Agent的soul.md不含调度逻辑** — 隔离职责，子Agent只管执行
7. **Reviewer必须是独立角色** — 不能与Orchestrator合并，否则评审失去意义
